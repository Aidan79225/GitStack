# Custom panel updates live on radio toggle

## Context

After landing the Custom panel pre-fill change (so swatches show the
currently-active theme's hex codes), there's still a UX gap: switching
the mode radio inside the dialog doesn't refresh the swatches. To
preview Light's tokens after looking at Dark's, the user has to Apply,
close the dialog, and re-open. That defeats the purpose of having all
four radios in the same dialog.

The fix: re-seed `_base_theme` and refresh swatches whenever the
selected radio changes.

## Scope

- **One Python file modified:** `git_gui/presentation/theme/manager.py`
  — add a public `theme_for_mode(mode)` method.
- **One Python file modified:**
  `git_gui/presentation/dialogs/theme_dialog.py` —
  `_on_mode_radio_toggled` re-seeds `_base_theme` and refreshes swatches
  to match the newly-selected mode.
- **One test file updated:**
  `tests/presentation/dialogs/test_theme_dialog.py` — one new test for
  the live-update behavior.

## Manager: new `theme_for_mode` API

`ThemeManager` already has the private `_resolve_theme()` helper which
maps `self._mode` to a `Theme`. Refactor it to a public
`theme_for_mode(mode: str) -> Theme` that takes the mode as an
argument:

```python
def theme_for_mode(self, mode: str) -> Theme:
    """Resolve a mode name to a Theme without changing the active mode."""
    if mode not in _VALID_MODES:
        raise ValueError(f"Invalid theme mode: {mode}")
    if mode == "light":
        return load_builtin("light")
    if mode == "dark":
        return load_builtin("dark")
    if mode == "custom":
        return self._load_custom_or_fallback()
    return self._system_theme()
```

The existing `_resolve_theme()` becomes a thin shim:
```python
def _resolve_theme(self) -> Theme:
    return self.theme_for_mode(self._mode)
```

Both `_load_custom_or_fallback` and `_system_theme` stay private — they
encode policy that's specific to the manager's lifecycle.

## Dialog: live refresh on radio toggle

`_on_mode_radio_toggled` currently only enables/disables the Custom
panel. Extend it to also re-seed `_base_theme` and refresh swatches:

```python
def _on_mode_radio_toggled(self, _checked: bool) -> None:
    mode = self._selected_mode()
    self._custom_panel.setEnabled(mode == "custom")
    if mode == self._base_theme_mode:
        return
    self._base_theme_mode = mode
    self._base_theme = self._mgr.theme_for_mode(mode)
    self._reset_to_base_state()
    for token, hex_value in self._working_colors.items():
        self._apply_swatch_color(token, hex_value)
    for i, hex_value in enumerate(self._working_lane_colors):
        self._apply_lane_swatch_color(i, hex_value)
```

The mode-equality short-circuit avoids redundant work:
`QButtonGroup.toggled` fires twice per click — once for the un-checked
radio, once for the newly-checked one — and both invocations of this
handler resolve to the same `_selected_mode()` (the newly-checked one).
The first invocation does the work; the second is a no-op.

Track `self._base_theme_mode` alongside `_base_theme` in `__init__` so
the comparison has something to compare against on the first call.

## Edge cases

- **Typography baseline must always be Dark.** Saved custom theme
  files store absolute pixel sizes generated by scaling some baseline's
  typography. To recover the slider value via division on reopen, the
  divisor must equal the multiplier — and historically the codebase
  used Dark as that baseline (`_write_custom_theme` originally read
  `dark.typography`). The previous task accidentally entangled the
  typography baseline with the swatch-source theme; this task
  separates them again.

  **Concrete change:** introduce `self._typography_base =
  load_builtin("dark")` set once in `__init__` and never updated.
  `_write_custom_theme` uses `self._typography_base.typography` for
  scaled-styles math; `_maybe_load_existing_custom_theme` uses it
  for the inverse division. `self._base_theme` continues to track the
  selected radio and is used only for swatch pre-fill / inherited
  non-edited Colors. The `__init__`-level "if mode == custom: use
  Dark" guard is removed because `_typography_base` now handles the
  round-trip concern independently of which radio is selected.

- **Custom radio doesn't re-seed.** The toggle handler short-circuits
  when `mode == "custom"`, so clicking the Custom radio leaves the
  swatches at whatever theme was last shown (Light, Dark, or System).
  Rationale: a user clicking Custom is opting into editing, and the
  most useful starting point is the theme they were just inspecting
  (the one whose values are visible right now). Yanking them to the
  saved custom file or to Dark would discard the inspection context.
  When the dialog first opens with mode already custom,
  `_maybe_load_existing_custom_theme` still loads the saved file —
  this short-circuit only affects mid-dialog radio toggles.

- **In-progress edits lost on non-custom radio toggle.** Acceptable
  for the inspection use case. If the user wanted to preserve edits
  across radio toggles, they'd have used Apply first. We won't add a
  confirm dialog or per-mode working state.

- **System mode color-scheme change while the dialog is open.** If the
  OS dark/light setting flips while the user is sitting on the System
  radio, the manager's `_on_system_scheme_changed` will refresh the
  active theme but the dialog's swatches won't auto-update. This is a
  rare edge; not addressed here.

## Verification

**Automated:**
```
uv run pytest tests/presentation/theme/test_manager.py -v
uv run pytest tests/presentation/dialogs/test_theme_dialog.py -v
uv run pytest tests/ -q
```

The existing manager tests must still pass after the
`_resolve_theme` → `theme_for_mode` refactor. The existing dialog tests
must still pass; the new `test_radio_toggle_refreshes_custom_panel`
test asserts that swatches change when the radio changes mid-dialog.

**Manual:**
1. `uv run python main.py`. Open the theme dialog while on Dark.
2. Click the Light radio. The Custom panel's swatches should
   immediately show Light's hex codes (`surface = #eff5fd`, etc.).
3. Click Dark. Swatches should switch to Dark's values.
4. Click System. Swatches should match whatever the OS setting
   resolves to (Light or Dark depending on your system).
5. Click Custom. If a saved custom file exists, swatches reflect it;
   otherwise they reflect Dark.
6. None of these clicks invokes Apply — the active theme outside the
   dialog is unchanged until the user clicks Apply.
